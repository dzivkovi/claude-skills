# Fix: Table Cell Text Descender Clipping in Word

## Problem

Table cell text in Word documents generated by the branded-docx skill had descenders (g, y, p, q) and digits (5, 9) visually clipped at the bottom. This was especially visible at non-100% zoom levels (110%, 130%) in Microsoft Word. The clipping did NOT appear in PDF exports.

## Root Cause

Two independent issues:

1. **Structural (fixable):** `docx-js` Paragraph objects inside table cells inherit Word's default paragraph after-spacing (~200 DXA / ~8pt). This invisible spacing consumes vertical space in the cell, pushing text against the bottom margin where descenders get clipped.
2. **Visual (unfixable display artifact):** Word's GDI text rendering engine rounds sub-pixel positions at fractional zoom levels. At 110% zoom, some character descenders lose their bottom pixel row. This is purely cosmetic and doesn't affect print or PDF output.

## Investigation

### Wrong path tried first

Increased cell margins from 80 to 100 DXA. This made rows taller but didn't fix clipping — the problem was paragraph spacing, not cell padding.

### XML forensic comparison

Unpacked both the clipped document and a known-good reference document to compare their XML. Key discovery in the reference document's table cells:

```xml
<w:spacing w:after="0" w:before="0"/>
```

And explicit vertical alignment:

```xml
<w:vAlign w:val="top"/>    <!-- data cells -->
<w:vAlign w:val="center"/> <!-- header cells -->
```

### Small test script

Created `test-table-clip.js` generating 3 table variants in one document:

- **A:** No fix (current — clipped)
- **B:** spacing:0 only
- **C:** spacing:0 + vAlign (reference approach)

Variant C matched the reference document's rendering.

### Zoom-dependent behavior

Discovered clipping is dynamic — appears at 110% zoom, disappears at higher zoom. PDF export never clips. This is Word's GDI pixel rounding, not a document structure issue. Do not chase this.

## Solution

Three changes to every table cell in docx-js code:

### 1. Add `spacing: { before: 0, after: 0 }` to cell paragraphs

```javascript
// Before (broken)
children: [new Paragraph({
  children: [new TextRun({ text: h, font: BRAND.heading, size: 18, bold: true, color: "FFFFFF" })]
})]

// After (fixed)
children: [new Paragraph({
  spacing: { before: 0, after: 0 },
  children: [new TextRun({ text: h, font: BRAND.heading, size: 18, bold: true, color: "FFFFFF" })]
})]
```

### 2. Add `verticalAlign` to table cells

```javascript
// Header cells
verticalAlign: VerticalAlign.CENTER,

// Data cells
verticalAlign: VerticalAlign.TOP,
```

### 3. Keep compact margins (80 DXA)

```javascript
margins: { top: 80, bottom: 80, left: 160, right: 160 },
```

The margins were temporarily increased to 100 during investigation — reverted back to 80 once the real fix (spacing:0) was identified.

### Import requirement

Add `VerticalAlign` to the docx-js import:

```javascript
const { ..., VerticalAlign, ... } = require('docx');
```

### Complete fixed `brandTable` pattern

```javascript
function brandTable(headers, rows, contentWidth = 9360) {
  const colWidth = Math.floor(contentWidth / headers.length);
  const border = { style: BorderStyle.SINGLE, size: 1, color: BRAND.lightGray };
  const borders = { top: border, bottom: border, left: border, right: border };

  return new Table({
    width: { size: contentWidth, type: WidthType.DXA },
    columnWidths: headers.map(() => colWidth),
    rows: [
      new TableRow({
        tableHeader: true,
        children: headers.map(h => new TableCell({
          borders,
          width: { size: colWidth, type: WidthType.DXA },
          shading: { fill: BRAND.dark, type: ShadingType.CLEAR },
          margins: { top: 80, bottom: 80, left: 160, right: 160 },
          verticalAlign: VerticalAlign.CENTER,
          children: [new Paragraph({
            spacing: { before: 0, after: 0 },
            children: [new TextRun({ text: h, font: BRAND.heading, size: 18, bold: true, color: "FFFFFF" })]
          })]
        }))
      }),
      ...rows.map((row, i) => new TableRow({
        children: row.map(cell => new TableCell({
          borders,
          width: { size: colWidth, type: WidthType.DXA },
          shading: { fill: i % 2 === 0 ? "FFFFFF" : BRAND.lightGray, type: ShadingType.CLEAR },
          margins: { top: 80, bottom: 80, left: 160, right: 160 },
          verticalAlign: VerticalAlign.TOP,
          children: [new Paragraph({
            spacing: { before: 0, after: 0 },
            children: [new TextRun({ text: String(cell), font: BRAND.body, size: 19, color: BRAND.dark })]
          })]
        }))
      }))
    ]
  });
}
```

## Prevention

### Checklist for table cells in docx-js

Every table cell paragraph must have:

1. **`spacing: { before: 0, after: 0 }`** on every `Paragraph` inside a `TableCell` — overrides Word's default after-spacing
2. **`verticalAlign`** on the `TableCell` itself — `CENTER` for headers, `TOP` for data cells
3. **Explicit `margins`** on `TableCell` — don't rely on table-level defaults
4. **Use descender-heavy test strings** when verifying: words like "Typography glyph" or "appy jiggly query" that expose clipping

### Testing strategy

Use the **comparison-variant approach** (the test-table-clip.js pattern):

- Generate a single .docx with multiple table variants stacked, each isolating one variable
- Test at multiple zoom levels in Word (100%, 110%, 125%, 67%)
- Always verify PDF export — it uses vector rendering and never clips

### Known Word rendering limitations (not bugs)

- **GDI pixel rounding at fractional zoom** — hairline clipping at certain zoom percentages is a Word display artifact. Disappears at 100% zoom and in PDF output. Do not attempt to fix this in document structure.
- **Font metric differences across platforms** — the same font may have slightly different ascender/descender metrics on Windows vs macOS. Use margins >= 40 DXA to absorb variance.

### Skill template as guardrail

The `brandTable()` function in SKILL.md is the canonical pattern. New table variants should copy it rather than building cells from scratch. Brand `.md` files control colors and fonts — the structural fix (spacing, vAlign) lives in the function template, independent of brand configuration.
